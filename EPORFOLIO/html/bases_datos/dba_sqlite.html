DBA_SQLITE_RA12

1. A mini-summary of what to do in the challenge:

This practice involved the complete administration of an SQLite database named "frutitas.db". The tasks were divided into three main blocks: storage analysis and performance optimization, where I analyzed query logs to identify that the "Mandarinas" table had high read traffic (227 SELECTs) and created indexes on color, size, and fecha_recogida to improve speed. Secondly, I performed backup and restoration operations (`frutitas_bonitas_bonitas.db`). Finally, I automated the backup process using a Bash script scheduled with Systemd timers to run daily at 23:59:59.

2. Two complex concepts that you have learned:

  • Log-Based Optimization: I learned how to analyze database logs to make data-driven decisions. Instead of guessing which indexes to create, I saw that the log showed a high frequency of SELECT queries on specific columns, justifying the creation of indexes to reduce search time.
  • Systemd Automation (Timer vs. Cron): I learned to use systemd units (`.service` and `.timer`) to automate tasks. Unlike a simple cron job, creating a specific service file allows for better control, logging, and dependency management for the backup script execution.

3. How have you felt:

I felt very professional automating the tasks. Moving from manually executing commands to setting up a "set and forget" system with Systemd made me feel like a real system administrator. Understanding the internals of SQLite, like the Journaling modes, was also very enlightening.

4. What would you improve about your solutions:

Currently, the database is protected by standard file permissions. To improve security for a production environment, I would implement SQLCipher to encrypt the database file itself. Additionally, regarding the PRAGMA settings, I would explore using PRAGMA journal_mode=WAL (Write-Ahead Logging) to improve concurrency, as discussed in the theoretical questions.

5. Technical words in English for the glossary:

  • PRAGMA: A special command in SQLite used to query or modify the internal operation of the library (e.g., page size, cache size).
  • WAL (Write-Ahead Logging): A journaling mode where changes are written to a separate log file first, allowing for faster and more concurrent reads and writes.
  • Index: A data structure (like a B-Tree) that improves the speed of data retrieval operations on a database table at the cost of additional writes and storage space.
  • Systemd Timer: A unit configuration file in Linux used to schedule tasks, acting as a modern alternative to cron.
  • Daemon: A background process that handles requests for services such as print spooling and file transfers, or in this case, the systemd manager handling the timers.

6. What grade would you give the practice statement and why (0 to 10), and what would you improve:

I would give this practice a 9 out of 10. It perfectly covers the lifecycle of a small database: creation, optimization, and maintenance. To achieve a 10, I would include a practical exercise on recovering a corrupted database or analyzing the performance difference before and after indexing with a timer.
